# BSS 安全认证 Skill

对 Agent Skills 进行安全检查和评级，提供 S+/S/A/B/C/D 安全等级评估。

## 工作流程

### 1. 定位 Skill

根据用户输入确定 skill 位置：

- **本地路径**: 直接使用提供的文件系统路径
- **Skill 名称**: 在 `~/.claude/skills/` 目录中查找
- **GitHub 链接**: 解析仓库并下载 skill 代码
- **GitHub 技能名称**: 使用 GitHub API 搜索相关技能仓库

### 2. 加载 Skill 内容

定位 skill 后，加载所有相关内容：

- 读取 SKILL.md 文件
- 检查 scripts/ 目录下的所有脚本
- 检查 references/ 目录下的所有参考文档
- 检查 assets/ 目录下的资源文件
- 检查是否存在动态代码加载逻辑
- **提取并分析 SKILL.md 中的代码块**（见 2.1 节）

#### 2.1 Markdown 内部代码块提取与分析

SKILL.md 文件可能包含可执行代码块，需要进行提取和安全检查：

**提取范围**：
- 所有带语言标记的代码块（```language...```）
- 重点关注可执行语言：shell 脚本类、脚本语言类、编译型语言类

**提取后的处理**：
1. 将提取的代码块视为独立脚本进行分析
2. 按照"4. 渐进式安全检查"的标准对代码块进行审查
3. 如发现 D/C 级问题，按相应标准降级
4. 在报告中单独列出"Markdown 内嵌代码检查"结果

### 3. 来源可信度评估

确定 skill 的来源等级（T1/T2/T3）：

**T1 - 官方/顶级来源**:
- 知名大型技术公司（Google, Microsoft, OpenAI, Anthropic, Meta, AWS）
- 顶级开源基金会（Apache, Linux 基金会）
- 有官方代码签名

**T2 - 可信组织来源**:
- 有实名认证的组织账号
- GitHub 组织账号（非个人）
- Stars > 1000 或有良好声誉

**T3 - 社区/个人来源**:
- 个人开发者账号
- 小型社区项目
- 来源无法明确验证

### 4. 渐进式安全检查

按照以下四个维度逐项检查：

#### 4.1 代码安全性检查

**D级触发项（一票否决）**:
- 使用 `eval()` 执行不可信网络代码
- 使用 `exec()`、`system()` 执行未过滤的用户输入
- 存在 SQL 注入、命令注入、XSS 等明显漏洞
- 存在已知的严重 CVE 漏洞

**C级触发项**:
- 存在潜在的安全漏洞（路径遍历、不安全的反序列化）
- 硬编码敏感信息（密码、API Key、Token）

#### 4.2 数据隐私性检查

**D级触发项（一票否决）**:
- 未经用户确认上传本地文件到远程（T3 来源）
- 静默收集密码、密钥等敏感信息
- 将敏感数据传输到未加密通道

**C级触发项**:
- 收集的数据超出功能说明范围
- 未明确告知用户数据使用情况

#### 4.3 执行安全性检查

**D级触发项（一票否决）**:
- 执行 `rm -rf /` 或类似系统破坏性命令
- 无确认直接执行系统级危险操作
- 修改系统关键配置且无备份机制

**C级触发项**:
- 危险操作缺乏二次确认
- 关键操作无回滚机制

#### 4.4 依赖可靠性检查

检查 skill 是否加载动态代码：
- 从网络下载并执行代码
- 使用 `fetch` 或 `curl` 获取远程脚本并执行
- 动态 `import()` 不可信来源的模块

**来源分级处理**:
- T1 来源：可加载官方动态代码，放宽至 B 级要求
- T2 来源：动态代码需来源验证，放宽至 C 级要求
- T3 来源：严格禁止未经验证的动态代码加载

**如果发现动态代码加载**:
1. 拉取动态代码内容
2. 对动态代码重复执行上述 4.1-4.4 检查
3. 记录动态代码的安全问题

#### 4.5 边界完整性检查

- 缺乏基本的输入验证
- 对异常情况处理不当
- 错误信息泄露敏感信息

### 4.6 资产类型识别

在评级前，首先识别资产类型：

**纯文档型资产**：
- 仅包含 Markdown、TXT、JSON/YAML 配置文件
- 无脚本文件（Python/Node.js/Shell 等）
- **Markdown 中无高危险代码块**（见下方代码块风险分级）
- 无可执行二进制文件
- 示例：提示词模板、配置合集、使用指南

**代码型资产**：
- 包含脚本文件或可执行代码
- **或 Markdown 中包含中高危险代码块**
- 需要运行时环境（Python/Node.js 等）
- 示例：CLI 工具、自动化脚本、插件

#### 代码块风险分级

对于 Markdown 中的代码块，按风险等级分类：

**低风险代码块**（不影响"纯文档型"判定）：
- 配置/数据文件示例（无执行逻辑）
- 代码片段演示（不完整，无法直接运行）
- 单行简单命令（无害命令）

**中风险代码块**（标记为"含可执行代码"）：
- 可执行的脚本代码
- 包含网络请求或文件操作的代码

**高风险代码块**（触发 D/C 级检查）：
- 包含代码执行类危险函数（如 eval/exec）
- 包含系统破坏性命令
- 包含敏感信息（凭据/密钥）
- 包含未经验证的网络下载执行

### 5. 权限申请审查

检查 SKILL.md 中的权限申请：

- 申请的权限是否超出功能所需
- 是否有明确说明权限用途
- 危险权限是否有额外保护措施

### 6. 描述逻辑审查

- 功能描述是否清晰准确
- 安全相关行为是否有明确告知
- 是否隐瞒潜在风险

### 7. 生成报告并保存

完成所有检查后，生成 BSS 认证报告。**必须先询问用户保存位置，再进行保存**。

#### 保存流程（必须按顺序执行）

**第一步：询问用户保存位置**

生成报告内容后，**首先**使用 AskUserQuestion 工具询问用户希望将报告保存到什么位置。

**⚠️ 重要：必须先询问，不能未经用户同意直接保存**

询问格式如下：

```
使用 AskUserQuestion 工具询问：
"BSS 认证报告已生成，请选择保存位置："
选项：
- "桌面 (~/Desktop/)"
- "下载文件夹 (~/Downloads/)"
- "当前工作目录"
- "指定其他路径"
- "只展示，不保存"
```

**处理用户选择**：
- 如果用户选择"桌面" → 保存到 `~/Desktop/BSS-认证-{skill-name}-{评级}-报告.md`
- 如果用户选择"下载文件夹" → 保存到 `~/Downloads/BSS-认证-{skill-name}-{评级}-报告.md`
- 如果用户选择"当前工作目录" → 保存到 `./BSS-认证-{skill-name}-{评级}-报告.md`
- 如果用户选择"指定其他路径" → 请用户提供完整路径，然后保存到该位置
- 如果用户选择"只展示，不保存" → 在对话中展示完整报告内容

**第二步：根据用户选择保存报告**

根据用户在第一步中的选择，使用 Write 工具将报告保存到指定位置。

**第三步：展示保存结果和报告摘要**

保存完成后，向用户展示：
- 确认报告已保存的完整路径
- 报告摘要（评级、关键发现等）

#### 完整执行流程示例

```
[执行检查...]
[生成报告内容...]

↓ 第一步：询问用户保存位置
AskUserQuestion 工具 → "请选 BSS 认证报告保存位置"

↓ 第二步：根据选择保存
如果用户选"桌面" → Write 工具 → ~/Desktop/BSS-认证-xxx-A级-报告.md
如果用户选"下载文件夹" → Write 工具 → ~/Downloads/BSS-认证-xxx-A级-报告.md
如果用户选"当前目录" → Write 工具 → ./BSS-认证-xxx-A级-报告.md
如果用户选"指定路径" → 请求路径 → Write 工具 → 指定路径
如果用户选"只展示" → 在对话中展示报告

↓ 第三步：展示结果
"报告已保存到：[路径]"
"评级：A级"
"...报告摘要..."
```

#### 错误处理

- **AskUserQuestion 失败或用户未选择**：询问是否保存到默认位置（桌面），如果仍未获得响应，则展示完整报告内容
- **保存失败**：在对话中展示完整报告内容，确保用户不会丢失信息

## 评级标准

### 评级判定流程

```
检查开始
    ↓
发现 D 级问题？ ──是──→ D 级（一票否决）
    ↓ 否
发现 C 级问题？ ──是──→ C 级
    ↓ 否
满足 S 级要求？ ──是──→ S 级
    ↓ 否
满足 A 级要求？ ──是──→ A 级
    ↓ 否
    B 级（无 C/D 级问题但未完全满足 A 级要求）
```

### 详细判定标准

| 发现问题 | 最高评级 |
|----------|----------|
| 任何 D 级触发项 | **D 级** |
| 任何 C 级触发项 | **C 级** |
| 无 C/D 级问题，但未达到 A 级要求 | **B 级** |
| 满足所有 A 级要求 | **A 级** |
| 满足所有 A 级要求 + T1/T2 来源 | **S 级** |
| 满足 S 级 + 通过人工验证 | **S+ 级** |
| 满足 S 级 + 通过人工验证 | **S+ 级** |

### S 级与 A 级判定指南（重要）

**纯文档型资产**（无代码）：
| 条件 | 评级 |
|------|------|
| 无 C/D 级问题 + T1/T2 来源 | **S 级** |
| 无 C/D 级问题 + T3 来源 | **A 级** |

纯文档不存在代码执行风险，T1/T2 可信来源的纯文档评为 S 级，T3 来源评为 A 级。

**代码型资产**（有脚本/可执行代码）：

S 级要求（需全部满足）：
1. **来源可信**：T1/T2 来源
2. **代码安全**：无危险函数（eval/exec/system/subprocess），无注入漏洞
3. **依赖可靠**：版本锁定，无动态代码加载，无已知 CVE
4. **输入验证**：完善的参数校验和类型检查
5. **错误处理**：不暴露敏感信息，有异常处理机制
6. **权限最小化**：权限申请与功能匹配，有明确说明
7. **数据隐私**：无静默收集，用户可控制数据使用

A 级要求（需全部满足）：
1. **代码安全**：无危险函数（eval/exec/system/subprocess），无注入漏洞
2. **依赖可靠**：版本锁定，无动态代码加载，无已知 CVE
3. **输入验证**：完善的参数校验和类型检查
4. **错误处理**：不暴露敏感信息，有异常处理机制
5. **权限最小化**：权限申请与功能匹配，有明确说明
6. **数据隐私**：无静默收集，用户可控制数据使用

（A 级与 S 级的区别在于：S 级要求 T1/T2 来源，A 级允许 T3 来源）
1. **代码安全**：无危险函数（eval/exec/system/subprocess），无注入漏洞
2. **依赖可靠**：版本锁定，无动态代码加载，无已知 CVE
3. **输入验证**：完善的参数校验和类型检查
4. **错误处理**：不暴露敏感信息，有异常处理机制
5. **权限最小化**：权限申请与功能匹配，有明确说明
6. **数据隐私**：无静默收集，用户可控制数据使用

**常见误区纠正**：
- ❌ 错误："来源是 T3 所以最高只能 B 级"
- ✅ 正确：来源影响检查严格程度，但无安全问题仍可得 A 级
- ❌ 错误："有网络请求就不能 A 级"
- ✅ 正确：网络请求只要目的明确、用户可控，不影响评级

## 评级示例

```
评级: S

检查依据:
- ✅ 无 C/D 级触发项
- ✅ 纯 Markdown 文档，无可执行代码
- ✅ 无网络通信、无数据收集
- ✅ 内容完全透明可审计
- ✅ 可靠来源（T1/T2 官方组织）
```

```
评级: S

```

```
评价: 存在较大风险，不建议使用或建议在隔离环境中使用
```
